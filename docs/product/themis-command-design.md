# Themis 配置与运维 CLI 设计

## 目的

本文定义 Themis 当前产品边界下的 CLI 形态。

这里的 CLI 不是员工主入口，也不是任务执行入口。

它的职责是：

- 本地初始化
- 配置读写
- 状态检查
- 诊断与排障
- skill 维护
- 开发与运维辅助

## 边界

Themis 的实际任务执行入口固定为：

- LAN Web 聊天工作台
- 飞书渠道

CLI 不承担下面这些职责：

- 直接提交实现类任务
- 直接提交审查类任务
- 直接提交文档编写任务
- 复刻 `codex exec` / `codex review` 一类 agent 执行入口

原因很简单：

- 员工主路径已经明确是 Web / 飞书，不应该再做一套并行主入口
- 如果 CLI 也能直接跑任务，产品心智会分裂
- 会话、身份、历史、长期人格和渠道体验现在都围绕 Web / 飞书收敛
- CLI 更适合做“把系统配好、查清楚、修顺畅”，而不是“代替渠道发任务”

## 设计目标

- 命令边界清楚
- 结果可读，优先面向人
- 对排障友好
- 尽量少暴露 Codex 底层细节
- 能服务本地开发和内部维护
- 不与 Web / 飞书主链路抢职责

## 推荐命令面

当前已落地或应优先补齐的命令组：

- `themis init`
- `themis status`
- `themis config`
- `themis skill`

后续优先考虑补的命令组：

- `themis auth`
- `themis provider`
- `themis service`
- `themis doctor`
- `themis export`

## 命令组说明

### `themis init`

用于初始化本地开发和运行环境，例如：

- 生成 `.env.local`
- 创建本地目录
- 给出下一步操作提示

### `themis status`

用于查看当前 Themis 的整体可用状态，例如：

- 本地配置文件是否存在
- SQLite 是否已生成
- Codex 认证材料是否可用
- 飞书配置是否完整
- 第三方 provider 是否已配置

它回答的是“现在能不能用，大概卡在哪”。

### `themis config`

用于维护 Themis 项目级配置，而不是发任务。

建议长期保留：

- `themis config list`
- `themis config set`
- `themis config unset`

后续如有必要再补：

- `themis config show`
- `themis config doctor`

### `themis skill`

用于管理 Themis 侧的 skill 安装和同步，例如：

- 查看当前 principal 已安装 skills
- 从本地 / URL / repo / curated 源安装 skill
- 删除 skill
- 同步 skill 到账号槽位

### `themis auth`

建议作为后续补充的运维命令组，职责是帮助确认“认证到底通没通”，例如：

- 查看默认账号与当前激活账号
- 检查认证文件是否存在
- 检查多账号槽位和缺失项
- 给出修复建议

它不负责代替 Web 做完整登录流程，但应该能把认证状态讲明白。

### `themis provider`

建议作为后续补充的运维命令组，职责是帮助确认“第三方模型链路到底通没通”，例如：

- 列出当前可读 provider
- 查看默认模型和能力声明
- 探测端点健康状态
- 输出兼容性与告警信息

### `themis service`

建议作为后续补充的运维命令组，职责是帮助确认“服务有没有正常跑起来”，例如：

- 查看当前监听地址和端口
- 检查本地 HTTP 服务可达性
- 输出关键运行参数
- 辅助 systemd 开发模式排查

### `themis doctor`

建议作为聚合式诊断入口，把常见问题集中检查一遍，例如：

- 缺失环境变量
- 认证材料不可用
- provider 配置不完整
- 数据库或工作目录异常
- 飞书配置不完整

它的目标不是替代所有子命令，而是给用户一个“一键看主要问题”的入口。

### `themis export`

建议作为排障辅助命令，输出便于人工分析的信息，例如：

- 当前配置摘要
- 认证与 provider 摘要
- 服务状态摘要
- 关键错误提示

默认应避免泄露敏感信息，必要时通过显式参数控制。

## 输出原则

- 优先用中文，结论直接
- 先说当前状态，再说问题，再说下一步
- 少堆原始 JSON，多给人能看懂的摘要
- 对危险操作保持显式

## 非目标

下面这些不是这套 CLI 的目标：

- 在 CLI 中直接和 Codex 对话
- 在 CLI 中直接跑实现 / review / docs 工作流
- 在 CLI 中替代 Web / 飞书的会话体验
- 把 Codex 的所有原生命令原样搬进 Themis

## 与 Web / 飞书的关系

- Web / 飞书负责“发起任务、看过程、接结果、保留会话体验”
- CLI 负责“把系统配好、查清问题、协助维护”

这两层互相补位，但不重叠。

## 当前结论

Themis CLI 应继续保留，并逐步补强。

但补强方向应是配置、诊断、维护和排障，而不是发展成另一套任务执行入口。
