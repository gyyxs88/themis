# Themis 局域网多节点硅基员工平台 / 分阶段落地计划

更新时间：2026-04-12 15:02 CST
文档性质：路线拆解稿。基于《Themis 局域网多节点硅基员工平台方案（V1 草案）》继续下收，回答“接下来应该按什么顺序推进、每阶段做到什么算完成”。

当前状态补充（2026-04-12）：

- `Phase 1 / 控制面底座` 已按当前计划范围完成，已落地控制面 `store` 抽象、SQLite/MySQL 原型、控制面门面，以及 `/api/platform/*` 最小平台 API 原型。
- `Phase 2 / 节点模型与调度租约` 的第一包也已完成：当前已固定 `node / execution_lease` 类型与 store 接口，SQLite 已补节点/租约表，平台 API 已新增 `/api/platform/nodes/register|heartbeat|list`，scheduler 也已接入最小节点匹配与 `execution_lease` 回填。
- 当前 Phase 2 的下一刀已切到节点治理补充：更具体地说，是 MySQL 节点/租约对齐、节点 TTL 过期后的 `offline` 语义，以及更明确的 `draining / 下线` 控制面动作。详细实施计划见独立文档。

## 1. 这份计划解决什么问题

前一份 V1 草案已经明确了目标形态：

- 数字员工是逻辑主体，不是“一人一个完整 Themis 实例”
- 平台负责统一控制面
- Themis 是入口形态之一
- 节点机器提供执行资源
- V1 先限定在公司局域网

接下来真正需要解决的，不再是“方向对不对”，而是：

- 先拆哪一刀
- 哪些能力必须先稳定
- 哪些能力应该后置
- 到哪一步才允许切主入口
- 什么时候才值得接入 Personal Themis

这份文档就是把这条路线拆成可执行阶段。

## 2. 适用范围与前提

这份计划只适用于当前讨论过的 V1 形态：

- 所有节点默认都在公司局域网内
- 有统一控制面与统一数据库
- 有一个公司级主 Themis 入口
- 有零个或多个 Personal Themis
- 有零个或多个 Worker Node

这份计划暂时不覆盖：

- 公网节点
- 多控制面高可用
- 去中心化共识
- 跨节点透明热迁移 thread/session

## 3. 路线拆分原则

### 3.1 先拆“状态归属”，再拆“执行位置”

当前最大问题不是“执行一定要跑在主 Themis 机器上”，而是：

- 状态和执行绑在一起
- 调度和入口绑在一起
- 平台事实和本地事实是同一个东西

所以路线顺序应该先把“控制面事实”抽出来，再把执行拆到节点上。

### 3.2 一次只改一条高风险边

这条路线里有三条高风险边：

- `store` 从本地 SQLite / registry 语义上收成平台控制面
- `run` 从本机执行上收成节点执行
- 主 Themis 从“宿主 + 入口”收成“入口 + 网关”

不建议把这三件事并在同一阶段里一起改。

### 3.3 先保住现有协作模型，不先推倒业务语义

当前代码里已经成型的业务对象：

- `managed_agent`
- `work_item`
- `mailbox`
- `handoff`
- `run`

这些语义已经能支撑创建、协作、治理和等待恢复，不应该在平台化第一刀里重写。第一阶段优先改的是“存在哪、谁来调度、谁来执行”，不是“业务对象全部改名重做”。

### 3.4 Gateway 切换必须后置

如果控制面和节点执行还不稳定，就不应该急着把飞书和 Web 主入口切过去。

原因很简单：

- 入口一旦先切，所有不稳定都会直接暴露给人类
- waiting / resume / 审计一旦不稳，主入口体验会迅速恶化

所以主 Themis 的 gateway 化必须建立在平台事实和节点执行都已经闭环之后。

## 4. 阶段总览

推荐按下面 6 个阶段推进：

### Phase 0：概念模型与非目标冻结

状态：已完成。

产出：

- V1 方案文档
- 推荐拓扑
- 对象关系图
- 调度时序图
- 非目标清单

进入下一阶段条件：

- 团队内部对“principal / node / control plane / gateway”的定义没有明显分歧

### Phase 1：控制面底座

核心问题：

- 把平台事实从单机 Themis 里抽出来

目标：

- 让 `managed_agent / work_item / mailbox / handoff / run` 具备中心化持久化入口
- 把当前服务层对底层持久化的依赖收敛成可替换接口
- 给后续 MySQL 控制面和平台 API 留出稳定边界

完成标志：

- 服务层可以不直接假定“事实一定在本地”
- 核心对象有统一 store 接口
- 能跑通最小的中心化读写闭环

### Phase 2：节点模型与调度租约

核心问题：

- 平台虽然有事实，但还不会把任务派到外部节点执行

目标：

- 建立 `node`、心跳、能力模型和 `execution_lease`
- 让调度器从“本地可跑”升级成“按节点能力选可跑位置”

完成标志：

- 平台能知道有哪些节点在线
- 平台知道节点还能接多少任务
- 平台能按工作区/凭据/槽位做最小匹配

### Phase 3：远端执行闭环

核心问题：

- 调度能选节点，不代表任务已经能在节点上稳定跑完

目标：

- 引入轻量 Worker Node
- 把 `run` 真正放到节点执行
- 打通 `queued -> running -> waiting -> resumed -> completed/failed` 全链

完成标志：

- 至少一类长期数字员工任务可以在非主 Themis 机器上稳定执行
- `waiting_human` / `waiting_agent` 能回到原节点恢复
- 节点失联后 lease 有明确恢复或中断策略

### Phase 4：主 Themis Gateway 化

核心问题：

- 当前主 Themis 仍然既是入口又是宿主

目标：

- 让主 Themis 保留飞书/Web 入口
- 但数字员工事实、治理事实和调度事实都从平台读取

完成标志：

- 主 Themis 通过平台 API 创建、派工、治理、查看状态
- 主 Themis 不再是数字员工真相来源

### Phase 5：Personal Themis 接入

核心问题：

- 个人助理可以成为主体，但权限和可见性边界尚未模型化

目标：

- 把 Personal Themis 作为平台主体接入
- 补齐 `visibility_scope / delegate_scope / workspace_scope / credential_scope`

完成标志：

- 个人助理可以进入公司层级
- 但不会天然获得全公司可见性和执行权限

### Phase 6：运维、灰度与扩面判断

核心问题：

- 即使技术链路通了，也不代表已经适合变成默认运行方式

目标：

- 固定部署方式、节点注册方式、监控、回滚和备份策略
- 明确是否值得继续扩到跨网节点或更强分布式

完成标志：

- 局域网 V1 具备最小可运维能力
- 对“要不要继续上公网 / 高可用 / 去中心化”有明确 go / no-go 结论

## 5. 各阶段详细拆解

### 5.1 Phase 1：控制面底座

这一阶段是整条路线最关键的一刀。

#### 为什么先做这个

如果控制面不先独立出来，后面所有多节点执行都会变成：

- 节点只是远程替主机跑一下
- 但事实仍然只有主 Themis 自己知道

那最终系统还是“单机平台 + 远端执行插件”，不是平台化。

#### 这一阶段要交付什么

- 统一的控制面对象边界
- 核心对象的 store 接口
- MySQL 原型 schema
- 平台 API 的最小读写闭环

#### 推荐首刀

1. 先定义平台事实边界。
   先回答清楚哪些对象必须进控制面：`organization`、`principal`、`node`、`work_item`、`run`、`execution_lease`、`mailbox`、`handoff`、`audit_log`。
2. 再定义 store 接口，而不是先写 MySQL SQL。
   先把 `ManagedAgentsService`、`ManagedAgentCoordinationService`、`ManagedAgentSchedulerService` 依赖的持久化能力按接口拆出来。
3. 最后做 MySQL 适配与最小平台 API。

#### 完成标准

- 核心服务层不直接依赖“本地 registry 是唯一事实来源”
- 新旧持久化实现至少有一层清晰边界
- 最小平台 API 能读写 `principal / work_item / run`

#### 这一阶段不要做什么

- 不要同时引入 Worker Node
- 不要在这一步就把 Web/飞书改成全部走平台
- 不要急着把所有老表一次性迁完

### 5.2 Phase 2：节点模型与调度租约

这一阶段解决的是“平台如何认识机器”。

#### 这一阶段要交付什么

- `node` 数据模型
- 节点注册与心跳
- 节点能力上报
- `execution_lease`
- 最小可用的节点调度策略

#### 推荐首刀

1. 先固定节点最小元数据：
   `node_id`、`status`、`slot_capacity`、`labels`、`workspace_capabilities`、`credential_capabilities`、`provider_capabilities`、`last_heartbeat_at`。
2. 再固定租约语义：
   一条 `run` 在某一时刻只能持有一个有效 `execution_lease`。
3. 最后才把调度器改成按节点匹配。

#### 完成标准

- 节点可以注册、续心跳、下线
- 调度器可以基于节点能力做最小可解释选择
- lease 超时后能把任务收口到明确状态

#### 这一阶段不要做什么

- 不做复杂负载均衡算法
- 不做跨节点热迁移
- 不做跨网节点接入

### 5.3 Phase 3：远端执行闭环

这一阶段开始真正回答“任务怎么离开主 Themis 机器去跑”。

#### 这一阶段要交付什么

- 轻量 Worker Node 进程
- 节点拉取或接收任务
- 节点回传运行状态
- waiting / resume 节点亲和性
- 节点失联后的租约恢复策略

#### 推荐首刀

1. 先只支持局域网节点执行一种最小任务类型。
   不要一开始就追求所有 managed agent 都能远端跑。
2. 优先复用现有 `AppServerTaskRuntime.sessionFactory` 和 `runtimeCatalogReader` 扩展点。
3. 先把 `waiting_human` 恢复链打通，再追求更多高级治理动作。

#### 完成标准

- 平台能把任务派到某个 Worker Node
- Node 能回传 `running / waiting / completed / failed`
- 人类回复后任务优先回到原节点继续执行

#### 这一阶段不要做什么

- 不做“隐式无感漂移”的恢复体验
- 不做节点本地目录自动同步成统一分布式文件系统

### 5.4 Phase 4：主 Themis Gateway 化

这一阶段是产品层面的切口，不是底层能力首刀。

#### 这一阶段要交付什么

- 主 Themis 调用平台 API
- Web `Agents` 面板读取平台事实
- 飞书治理动作写入平台
- 主 Themis 只保留入口与视图职责

#### 推荐首刀

1. 先切读路径，再切写路径。
   先让总览、详情、治理面读取平台事实。
2. 再切创建、派工、回复 waiting 这三条关键写路径。
3. 最后收掉主 Themis 的本地真相来源角色。

#### 完成标准

- 主 Themis 断开本地控制面后仍能看见完整组织级状态
- 人类从飞书/Web 发起的治理动作都能经平台生效

#### 这一阶段不要做什么

- 不重写主 Themis 的产品壳
- 不在这一步同时引入 Personal Themis

### 5.5 Phase 5：Personal Themis 接入

这一阶段不是“功能锦上添花”，而是权限边界题。

#### 这一阶段要交付什么

- Personal Themis 主体类型
- 可见性边界
- 委派边界
- 工作区边界
- 凭据边界

#### 推荐首刀

1. 先支持“个人可见 + 可被公司主入口治理”的保守模型。
2. 再逐步放开更复杂的双向委派。

#### 完成标准

- 个人助理可作为主体进入层级
- 但默认不会读取或接触不属于自己的组织资产

#### 这一阶段不要做什么

- 不做默认全公司互相可见
- 不做无边界凭据复用

### 5.6 Phase 6：运维、灰度与扩面判断

这一阶段是把“能跑”变成“可长期运营”。

#### 这一阶段要交付什么

- 控制面部署指南
- 主 Themis 部署指南
- Worker Node 部署指南
- 节点注册与下线流程
- 监控项与告警项
- 回滚和备份方案

#### 推荐首刀

1. 先固定局域网部署拓扑。
2. 再固定监控和回滚。
3. 最后再讨论要不要继续做跨网节点。

#### 完成标准

- 局域网内新增一台 Worker Node 有标准流程
- 主 Themis 和平台控制面故障有明确处置路径
- 团队对“是否继续扩面”有清晰判断标准

## 6. 当前推荐首刀顺序

如果只看接下来最值得开的三刀，我推荐是这三个：

### 第一刀：控制面对象与 store 接口定型

目标：

- 固定平台事实对象
- 固定服务层与存储层边界

原因：

- 这是后续 MySQL、平台 API、节点调度的共同前提

交付物：

- 控制面对象草图
- store 接口草图
- 当前服务层依赖映射

### 第二刀：`node + execution_lease` 模型首版

目标：

- 固定“机器如何被平台识别”和“运行如何绑定节点”

原因：

- 不把这层钉住，后面远端执行很容易语义漂移

交付物：

- `node` 最小字段
- `execution_lease` 状态机
- 调度器最小选择规则

### 第三刀：远端执行最小验证链

目标：

- 证明至少一种 managed agent 工作流可以在非主 Themis 节点执行

原因：

- 只有证明执行真能离开主机，平台化才不是纸上架构

交付物：

- 轻量 Worker Node 原型
- 最小任务拉取与回传链
- waiting / resume 节点亲和性验证

## 7. 阶段切换检查点

每进入下一阶段前，都建议做一次 go / no-go 判断。

### Checkpoint A：Phase 1 结束后

要确认：

- 控制面对象边界是否稳定
- store 接口是否能覆盖现有主链路
- MySQL 原型是否没有把服务层改成一团耦合

如果这里不稳，就不要进入节点执行阶段。

### Checkpoint B：Phase 3 结束后

要确认：

- waiting / resume 是否稳定
- 节点失联后的 run 收口是否可解释
- 审计事实是否完整

如果这里不稳，就不要切主 Themis gateway。

### Checkpoint C：Phase 4 结束后

要确认：

- 主入口是否已完全依赖平台事实
- 人类侧治理体验是否没有明显倒退

如果这里不稳，就不要开放 Personal Themis。

## 8. 当前不建议提前开的题

下面这些题都容易让路线失焦，当前不建议提前开：

- 去中心化控制面
- 公网节点
- 跨节点热迁移会话
- 自动分布式工作区同步
- 个人 Themis 默认进入全公司互通
- 复杂负载均衡和调度优化
