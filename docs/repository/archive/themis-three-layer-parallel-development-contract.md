# Themis 三层拆仓开发约定

## 结论

- 平台层、主 Themis、Worker Node 现在按独立系统理解，不再按“同一仓库里的三个子目录”理解。
- 三层不仅部署上应独立，代码仓、组件、发布节奏、运行态目录也都不应继续共用。
- 当前仓库里仍然混放的三层代码，只能视为历史过渡形态；后续不应再以“先共用、以后再拆”为默认开发方式。

## 为什么现在要把话说死

- 三层正常落地就是三台主机，部署边界本来就不是一个进程、一个目录、一个发布包。
- 平台层的关注点是控制面、调度、租约、值班和审计；主 Themis 的关注点是人类入口、主 Agent 主责判断与协作；Worker 的关注点是本机执行。
- 这三层的权限面、失败模式、升级节奏和值班责任都不同，继续共用代码仓和组件，只会把边界重新揉回去。
- 尤其是页面层，如果继续共用前端组件和页面模块，很容易把“平台后台”和“Themis 自己的页面”重新做成一锅。

## 硬规则

- 不共用代码仓。
- 不共用前端页面与页面级组件。
- 不跨层直接 import 对方业务代码。
- 不共用构建产物、运行态目录或发布流水线。
- 三层之间只通过显式契约联调，不通过“顺手复用一段源码”联调。

## 三层分别负责什么

### 1. 平台层

负责：

- 全局 control plane
- `/api/platform/*`
- 节点、租约、调度、回收、审计
- 平台管理员 / 值班页面
- 平台侧鉴权、监控、备份恢复

不负责：

- Themis 面向人类的对话页面
- 主 Themis 的主责判断
- Worker 的本机执行细节

### 2. 主 Themis

负责：

- Web / 飞书等人类入口
- 主 Agent 的对话、任务理解、拆解、派工
- 当前 Themis 自己的任务、历史、运行参数、身份配置
- 必要时自己执行

不负责：

- 全局节点值班面板
- 全局租约 / 调度后台
- Worker 节点治理页面

### 3. Worker Node

负责：

- `register -> heartbeat -> pull -> execute -> report`
- 本机工作区、credential、provider 能力声明
- 本机预检与运行边界校验
- 节点值守命令与节点本机运行

不负责：

- 人类侧页面
- 平台控制面真相源
- 主 Themis 的协作与治理页面

## 允许共享的东西，只能是契约

允许共享的不是源码，而是契约：

- 平台 API 契约
- Worker 协议契约
- 项目连续性 / 工作区绑定契约
- 鉴权与错误码契约

推荐做法：

- 用文档、OpenAPI、JSON Schema、事件协议文档或生成产物表达契约。
- 如果确实需要 SDK，也应由契约生成，不应直接引用另一个业务仓的源码目录。

## 对页面和组件的直接约束

- 平台页面只属于平台层。
- Themis 页面只属于主 Themis。
- Worker 默认不提供人类侧页面。
- 不允许再出现“先挂在 Themis 页面里，等以后再迁到平台”的默认路径。
- 如果某个页面展示的是全局 `nodes / execution leases / scheduler / oncall`，那它从第一天起就应该归平台层。

## 对当前仓库的影响

- 当前仓库里已有的“同仓三层”内容，不应再继续扩张。
- 后续如果还要推进三层演进，优先做拆分，而不是继续在当前仓库里补新的跨层页面或共享组件。
- 当前仓库里的旧实现如果必须改，只能按“迁移存量”心态处理，不能再把它当成长期正确形态。

## 以后怎么判断一个需求该落哪层

- 如果问题是“谁能执行、节点是否健康、租约怎么回收、值班怎么处理”，落平台层。
- 如果问题是“这个 Themis 自己怎么看、怎么聊、怎么派工、怎么展示自己的状态”，落主 Themis。
- 如果问题是“某台节点本机怎么执行、怎么上报、怎么预检”，落 Worker。
- 如果一个需求天然横跨三层，先改契约，再各层分别实现，不要回到单仓共享代码。
