## 目标

在不打散现有文本主链路的前提下，先把最值得卡片化的行为收成一轮最小 PoC 草案，回答三个问题：

1. 哪些动作真的适合做成卡片。
2. 卡片应该承载哪些字段和按钮。
3. 如果回调失败、token 过期或更新次数耗尽，怎么安全退回文本链。

## 前提

这份草案只在下面前提满足后才适合进入 PoC：

- 文本态主链路已经稳定收口。
- 真实飞书低成本复跑已经从“进行中”移出。
- 团队接受飞书卡片回调的额外约束：`3` 秒同步响应、更新 token 仅 `30` 分钟有效、每张卡最多更新 `2` 次。

如果这些前提没有满足，优先继续维护文本链路，而不是先做卡片。

## 设计原则

- 卡片只承载“短平快、单次决策、状态明确”的动作，不承载长文本对话。
- 每个按钮都必须有清晰的降级退路；退不回文本链或 Web 管理面的动作，不进入第一轮 PoC。
- 卡片只做少量高价值动作，不追求把现有所有飞书命令都包成按钮。
- 卡片只消费服务端事实，不在飞书渠道单独维护第二套任务真相。
- 同一张卡的更新预算要保守使用，默认把“一次点击”控制在“一次处理中更新 + 一次最终更新”以内。

## 第一轮建议做成卡片的行为

### 1. 运行中 / 阻塞状态卡

#### 适用场景

- 任务处于 `running`、`waiting_human`、`waiting_agent`、`blocked`、`failed`。
- 当前需要的是“看懂现在卡在哪”和“快速跳到下一步”，不是继续多轮输入。

#### 为什么值得先做

- 这是文本态最容易变长、最难扫读的一类信息。
- 它能补足飞书移动端“状态可视性”短板，但不会抢走文本链路的主导权。

#### 建议字段

- 任务标题：优先显示当前 turn goal，没有则显示最近摘要。
- 状态标签：`运行中 / 等人回复 / 等 agent / 阻塞 / 失败`。
- 当前会话：`sessionId` 或可读摘要。
- 最近动作时间：避免用户分不清是不是旧状态。
- 当前阻塞原因：一行摘要。
- 下一步建议：例如“请先审批”“请直接回复文本”“请切到 Web 查看详情”。
- 文本降级提示：明确给出现有命令，如 `/approve <actionId>`、`/deny <actionId>`、`/reply <actionId> <内容>`、`/current`。

#### 第一轮按钮

- `刷新状态`
- `查看下一步`
- `打开 Web 详情`

#### 第一轮不放的按钮

- `取消任务`
  原因：当前飞书侧没有稳定的文本降级命令，直接放进第一轮会把中断语义、幂等和错误口径一起拉进来。
- `新开会话`
  原因：当前会话切换和群聊管理员语义更重，先不要把它塞进状态卡。

#### 降级策略

- 如果回调失败或 token 过期，直接发一条文本摘要：
  - 当前状态
  - 现有可用命令
  - 如有必要，提示去 Web 继续

### 2. 单步审批卡

#### 适用场景

- `approval` waiting action。
- 动作本质上是一次二元决策，不需要长文本解释。

#### 为什么值得先做

- 它和卡片按钮天然匹配，收益最高，风险最低。
- 现有文本降级链已经成熟：`/approve`、`/deny`。

#### 建议字段

- 审批标题：例如“需要确认一项操作”。
- 审批摘要：本次要批准什么。
- 影响范围：工作区、网络、外部账号或目标资源。
- 来源会话：当前会话摘要。
- `actionId`
- 文本降级命令：`/approve <actionId>`、`/deny <actionId>`

#### 第一轮按钮

- `批准`
- `拒绝`
- `打开 Web 详情`

#### 明确不做

- 不把 `user-input` 自由文本补充也塞进同一张卡。
- 不把 `review / steer` 升级成按钮卡；它们仍保留文本输入或 Web 入口。

#### 降级策略

- 卡片提交成功但更新失败：发文本“审批已收到，正在处理”。
- 回调超时、按钮重复点击、token 过期：发文本并明确提示继续用 `/approve` 或 `/deny`。

### 3. Agent 治理卡

#### 适用场景

- 持久化 agent 出现 `waiting_human`、`waiting_agent`、`handoff_pending`、mailbox 待确认。
- 当前目标是做最小治理动作，而不是把 agent 变成一个长期卡片聊天窗口。

#### 为什么值得做

- Themis 现在已经有组织级汇总台、mailbox ack、pause / resume、handoff 事实。
- 这些动作本质上是管理动作，比自由对话更适合按钮。

#### 建议字段

- agent 名称
- 当前状态：`active / paused / degraded / waiting_*`
- 当前 work item 标题
- 最近 blocker / handoff 摘要
- 最近活动时间
- 推荐动作
- Web 管理面提示：告诉用户复杂操作去 `Agents` 面板

#### 第一轮按钮

- `确认消息`
  适用于 mailbox ack。
- `暂停 agent` 或 `恢复 agent`
  按当前状态二选一显示。
- `打开 Web 管理面`

#### 第一轮先不做成按钮的治理动作

- `接管后直接在卡片里连续指挥`
- `在卡片里填写长文本治理回复`
- `在飞书里做复杂多 agent 编排`

这些动作都更适合 Web `Agents` 面板，而不是第一轮卡片 PoC。

#### 降级策略

- 如果卡片动作失败，直接发送文本摘要，并提示去 Web `Agents` 面板继续。
- mailbox ack 失败时要明确告诉用户“当前消息还未确认”，不要默默吞掉。

## 明确不建议卡片化的行为

- `user-input` 自由文本补充
- `review / steer`
- `/use`、`/workspace`、`/group`
- 长任务的流式正文展示
- 复杂群协作编排

原因一致：这些行为要么依赖长文本，要么依赖复杂上下文切换，要么依赖管理员语义，不适合塞进第一轮卡片。

## 建议回调链

### 基本流程

1. 飞书收到 `card.action.trigger`。
2. 服务端按按钮类型路由到对应处理器。
3. 先做鉴权、幂等和 card binding 查找。
4. `3` 秒内同步返回“已收到，处理中”。
5. 异步调用 Themis 现有服务层能力。
6. 尝试更新原卡；如果更新失败，则发文本 follow-up。

### 第一轮建议的处理器拆分

- `src/channels/feishu/feishu-card-renderer.ts`
  负责把服务端事实渲染成卡片 payload。
- `src/channels/feishu/feishu-card-callback.ts`
  负责 `card.action.trigger` 路由、鉴权、幂等和同步响应。
- `src/channels/feishu/feishu-card-actions.ts`
  负责把按钮动作映射到现有服务层能力。
- `src/channels/feishu/feishu-card-store.ts`
  负责卡片绑定、callback token、更新次数和最近一次动作结果。

### 第一轮最小持久化事实

- `cardKey`
- `messageId`
- `sessionId`
- `conversationId`
- `ownerPrincipalId`
- `cardType`
- 目标对象 id：如 `actionId`、`workItemId`、`mailboxEntryId`
- `callbackToken`
- `tokenExpiresAt`
- `updateCount`
- `lastCallbackAt`
- `lastActionDedupKey`

## 更新预算策略

飞书卡片更新预算很紧，第一轮要保守：

- 一次点击默认最多消耗 `2` 次更新：
  - 第一次：同步“处理中”
  - 第二次：最终结果
- 如果当前卡已经接近更新上限，就不要再强行刷新原卡，直接走文本 follow-up。
- 不把长任务流式状态写成持续刷新卡片；状态卡只做快照，不做流式播放器。

## 幂等与失败处理

### 幂等键建议

- `messageId + actionValue + operatorId + actionTimestampBucket`

### 必须覆盖的失败场景

- 重复点击
- 旧卡延迟回调
- token 过期
- 更新次数耗尽
- 目标对象已经结束：如审批已处理、mailbox 已 ack、agent 已 paused
- 群聊里操作者没有权限

### 统一失败口径

- 卡片动作失败时，不要只写“失败了”。
- 必须回给用户：
  - 为什么失败
  - 当前对象现在是什么状态
  - 还能怎么继续

## 与现有文本链的关系

- 卡片是提效层，不是主链。
- 文本命令链必须继续保留，尤其是：
  - `/approve`
  - `/deny`
  - `/reply`
  - `/current`
- 任何一张卡片失效后，用户都能回到文本命令或 Web 管理面，不允许出现“只能点卡片、卡片坏了就无路可走”的状态。

## 建议实施顺序

1. 只做审批卡
   原因：文本降级最完整，风险最低。
2. 再做运行中 / 阻塞状态卡
   原因：提升移动端可视性，但不改变主链语义。
3. 最后做 agent 治理卡
   原因：它的价值高，但涉及治理权限、mailbox、manager 汇总，语义更重。

## PoC 成功标准

- 审批卡能稳定替代 `/approve` / `/deny` 的一部分高频场景。
- 状态卡能显著减少“现在到底卡在哪”的追问。
- agent 治理卡能覆盖最小治理动作，但没有把飞书变成第二个复杂控制台。
- 任一回调失败后，用户仍能回到文本链或 Web 管理面继续完成动作。

## 当前结论

- 现在最值得卡片化的，不是“所有飞书行为”，而是：
  - 运行中 / 阻塞状态卡
  - 单步审批卡
  - agent 治理卡
- 其中最适合先开 PoC 的，是审批卡。
- 如果只允许做一张卡，优先做审批卡；如果允许做两张，再补状态卡；agent 治理卡放第三顺位。
